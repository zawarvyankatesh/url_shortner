# AI-Powered Kubernetes Incident Response Agent (AIOps)

An end-to-end AIOps project: a sample application runs on Kubernetes with full
monitoring and alerting, and an **AI agent** automatically investigates alerts —
gathering read-only cluster context, asking an LLM for a root-cause analysis, and
emailing the report to the on-call engineer.

When something breaks, the on-call engineer receives:
1. an immediate **"alert received – investigating"** email, and
2. a follow-up **root-cause analysis** email (summary, root cause, evidence,
   suggested remediation, confidence) generated by an LLM from live evidence.

**End-to-end sequence**

1. A commit triggers **AWS CodeBuild**, which tests, builds multi-stage Docker
   images for the app and the agent, and pushes them to Docker Hub.
2. The app + Redis run on Kubernetes; the app exposes Prometheus metrics.
3. **Prometheus** scrapes the app (via pod annotations) and **kube-state-metrics**,
   and evaluates alert rules.
4. When a rule fires, **Alertmanager** routes the alert via webhook to the agent.
5. The **agent** emails a "received" notice, collects read-only evidence
   (pod status, events, current/previous logs, Prometheus signals), sends it to
   the LLM, and emails the root-cause analysis.

## What we built (phase by phase)

1. **Application** – a FastAPI URL shortener backed by Redis, with Prometheus
   metrics and separate liveness (`/healthz`) and readiness (`/readyz`) probes.
2. **CI/CD** – an AWS CodeBuild `buildspec.yml` that runs tests and builds/pushes
   non-root, multi-stage Docker images (tagged `latest` + short git SHA) for both
   the app and the agent.
3. **Kubernetes deployment** – Namespace, Deployment (2 replicas), Service,
   Redis, and a ConfigMap; probes wired to the health endpoints.
4. **Monitoring & alerting** – a self-contained stack (Prometheus, Alertmanager,
   kube-state-metrics) with annotation-based scraping and alert rules for
   `RedisDown`, `PodCrashLooping`, `PodNotReady`, `ContainerWaiting`
   (ImagePull/CrashLoop), `PodOOMKilled`, and high 5xx error rate.
5. **AI incident-analysis agent** – a FastAPI service that receives Alertmanager
   webhooks and runs `email → collect (read-only) → LLM → email`.

## Security & guardrails

- **Read-only RBAC**: the agent's ServiceAccount can only `get`/`list` a limited
  set of resources cluster-wide. It cannot create, update, delete, exec, or read
  Secrets. Even a misbehaving model or a prompt-injection attempt in a log line
  has no write capability to abuse.
- **The LLM has no tools and no cluster access** – it only reasons over evidence
  the agent already gathered.
- **Human-in-the-loop**: all remediations are suggestions for review; the agent
  never executes changes.
- **Secrets** (LLM token, SMTP password) live in a Kubernetes Secret created
  imperatively, never committed to git.

## Deployment (overview)

```bash
# 1. App + Redis
kubectl apply -f k8s/namespace.yaml
kubectl apply -f k8s/

# 2. Monitoring
kubectl apply -f k8s/monitoring/

# 3. Agent (create the Secret first - see k8s/agent/SECRET_SETUP.md)
kubectl apply -f k8s/agent/00-namespace.yaml
kubectl create secret generic incident-agent-secrets -n ai-agent \
  --from-literal=LLM_API_KEY='<your-llm-token>' \
  --from-literal=SMTP_PASSWORD=''
kubectl apply -f k8s/agent/
kubectl apply -f k8s/monitoring/06-alertmanager-config.yaml
kubectl rollout restart deployment/alertmanager -n monitoring
```

Set your environment-specific values (LLM proxy endpoint, SMTP relay, recipient)
in `k8s/agent/02-configmap.yaml` before applying.

### Trigger a test incident

```bash
kubectl scale deployment redis -n url-shortener --replicas=0   # fires RedisDown
# ... agent emails an RCA ...
kubectl scale deployment redis -n url-shortener --replicas=1   # recover
```

## Issues faced & how we solved them

A record of the real problems hit while building this, and the fixes — useful for
anyone reproducing it.

### 1. `port-forward` "connection refused"
**Symptom:** `curl localhost:8080` failed with connection refused.
**Cause:** `kubectl port-forward` is a foreground process; it had been stopped
(Ctrl+C) before the curl ran.
**Fix:** keep the port-forward running in one terminal and curl from another, or
background it. The tunnel only exists while the command runs.

### 2. Short-URL redirect showed "Could not resolve host"
**Symptom:** hitting a short link returned a correct `307` with the right
`Location`, then curl failed to resolve the destination.
**Cause:** the app correctly returns a redirect; the *server* had no external DNS
to then reach the destination. Not an app bug.
**Fix:** none needed — verified via the returned `Location` header. A browser with
internet follows the redirect normally.

### 3. Dashboards (Prometheus UI) not reachable on the POC server
**Symptom:** `http://localhost:9090` wouldn't open; server ports were blocked.
**Cause:** the UI ports are firewalled on the POC host.
**Fix:** the project doesn't need a UI — verified everything via the Prometheus
HTTP API with `curl` from the server, and used an SSH tunnel
(`ssh -L 9090:localhost:9090 ...`) when a browser view was wanted.

### 4. Building the agent image failed with "Network is unreachable"
**Symptom:** `docker build` failed at `pip install` (`No matching distribution`),
retrying to reach PyPI.
**Cause:** the POC server has no internet egress, so `pip` couldn't download
packages during the build.
**Fix:** build in CI instead. Extended the CodeBuild `buildspec.yml` to build and
push **both** the app and the agent images (CodeBuild has internet); the cluster
just pulls the finished images from Docker Hub.

### 5. LLM call returned `404 Invalid token`
**Symptom:** `openai.NotFoundError: 404 - Invalid token`.
**Cause (two parts):**
  - the client wasn't sending the caller identity the proxy expected, and
  - the internal host was being routed through a corporate HTTP proxy.
**Fix:** send the caller as an `X-Effective-Caller` header, set `NO_PROXY` for the
LLM host so the request goes direct, and keep TLS verification off for the
self-signed internal cert.

### 6. Secret kept showing `REPLACE_ME`
**Symptom:** after creating the real Secret, the pod's token was `REPLACE_ME`.
**Cause:** a template `03-secret.example.yaml` lived in `k8s/agent/`, so
`kubectl apply -f k8s/agent/` applied it too and **overwrote** the real Secret
with the placeholder.
**Fix:** removed the applyable template and moved the instructions to
`SECRET_SETUP.md`; the Secret is now created only imperatively and can't be
clobbered by `apply`.

### 7. In-cluster agent shares fate with the cluster (design note)
An agent running inside the cluster it monitors can't investigate if the cluster
control plane or its own node is down. Accepted for this project (simplicity,
self-contained); in production the agent would run in a separate failure domain.
